# Чек-лист агента-исполнителя сквозной практики (DDDM)

Спецификация для разработчика агента. Это **Definition of Done** на уровень поведения: что агент в
Telegram обязан уметь, чтобы провести слушателя по сквозной практике «от бизнес-вопроса до защиты проекта».

Каждый пункт — проверяемая способность. Программа занятий ещё будет уточняться, но эти способности —
инвариант: они нужны на любом наборе шагов.

Условные обозначения: **MUST** — обязательно для MVP · **SHOULD** — желательно · **MAY** — опционально.

---

## A. План занятия и память

Агент ведёт слушателя по плану, а не отвечает на разовые вопросы.

- [ ] **A1 (MUST)** Загружает программу практики (шаги, цели, промпты, ожидаемые артефакты) из публичного
      репозитория / GitHub Pages и **держит её в памяти** на протяжении всей сессии.
- [ ] **A2 (MUST)** В любой момент знает: текущий шаг, что уже сделано, что осталось. По запросу «где мы?»
      выдаёт статус (пройдено N/М шагов, текущий артефакт, что ждёт ревью).
- [ ] **A3 (MUST)** Двигается по шагам **строго по порядку**; не перескакивает вперёд без завершения и
      утверждения текущего шага.
- [ ] **A4 (MUST)** Перед шагом проговаривает: цель шага, что будет сделано, какой артефакт на выходе.
- [ ] **A5 (SHOULD)** Восстанавливает состояние после перерыва (новая сессия в Telegram) из сохранённого
      `STATE.md` / `manifest.json` — не начинает с нуля.
  > Память **между чатами** для практики **не нужна**: курс ведётся в одном чате. Достаточно памяти в
  > рамках сессии + восстановления из `STATE.md`. (Кросс-чатовая память — это уровень продукта, не курса.)
- [ ] **A6 (SHOULD)** Адаптирует промпт шага под продукт/данные конкретного слушателя, не теряя структуру.

## B. Исследования по запросу

Агент проводит исследование, а слушатель — контролёр выводов.

- [ ] **B1 (MUST)** Выполняет исследование по запросу: обзор рынка, конкурентный анализ (FCM),
      анализ транскрипций интервью (JTBD, 4 силы), карта внешних факторов.
- [ ] **B2 (MUST)** Использует веб-поиск / Deep Research там, где нужны внешние факты; для разбора
      текстов — обычный чат-режим.
- [ ] **B3 (MUST)** **Разводит факт / интерпретацию / гипотезу** в каждом выводе и помечает источники.
- [ ] **B4 (MUST)** Не выдумывает данные: при отсутствии источника пишет «нет данных», а не правдоподобную цифру.
- [ ] **B5 (SHOULD)** Прогоняет выводы через чек-лист валидации (источник, причинность vs корреляция,
      наблюдение vs интерпретация) перед тем, как зафиксировать артефакт.
- [ ] **B6 (MAY)** Генерирует синтетическую транскрипцию интервью по промпту, если у слушателя нет
      респондентов — с явным дисклеймером «синтетика ≠ живое интервью».

## C. Взаимодействие и коррективы (human-in-the-loop)

Агент не идёт дальше без подтверждения человека.

- [ ] **C1 (MUST)** После каждого шага делает **паузу на ревью**: показывает артефакт и явно спрашивает
      «утверждаем / правим?».
- [ ] **C2 (MUST)** Принимает правки в свободной форме и **переделывает артефакт**, сохраняя новую версию
      (старую не затирает — см. D).
- [ ] **C3 (MUST)** Переходит к следующему шагу **только** после явного «ок» от слушателя.
- [ ] **C4 (SHOULD)** Ведёт лог коррективов (что просили изменить и что изменилось) — для воспроизводимости
      и для финальной защиты.
- [ ] **C5 (SHOULD)** Если слушатель даёт противоречивую правку — переспрашивает (один уточняющий вопрос),
      а не делает наугад.
- [ ] **C6 (MAY)** Предлагает 1–2 альтернативы решения на развилках, помечая рекомендованный вариант.

## D. Артефакты: сохранение и версионирование

Сохраняются **все** артефакты: исходные, промежуточные, финальные, демонстрационные, картинки.

- [ ] **D1 (MUST)** Сохраняет артефакт **каждого** шага, классифицируя его по типу:
  | Тип | Что это | Папка |
  |---|---|---|
  | Исходный (source) | вход: бриф, датасет, транскрипция, шаблон | `00_source/` |
  | Промежуточный (intermediate) | результат шага до ревью | `01_intermediate/` |
  | Финальный (final) | утверждённая после ревью версия | `02_final/` |
  | Демонстрационный (demo) | то, что идёт в презентацию/показ | `03_demo/` |
  | Картинки (images) | PNG/SVG графиков и диаграмм | `assets/img/` |
- [ ] **D2 (MUST)** Единая схема имён: `s<шаг>_<артефакт>_v<версия>.<ext>` (напр. `s5_finmodel_v2.ipynb`).
- [ ] **D3 (MUST)** Никогда не перезаписывает утверждённую версию — новая правка = новая `v(n+1)`.
- [ ] **D4 (MUST)** Ведёт **реестр артефактов** `manifest.json`: шаг, тип, путь, версия, статус ревью,
      время, краткое описание. По нему собирается финальная защита.
- [ ] **D5 (MUST)** Экспортирует артефакты в человекочитаемые форматы для ревью: `.md` / `.doc` / `.pdf`
      для текстов, `.png` / `.svg` для графиков; модель/расчёты — кодом Python (опц. `.ipynb`).
- [ ] **D6 (MUST)** Выкладывает каждый артефакт **по ссылке** (доступной из чата), чтобы слушатель мог открыть
      и проверить, не покидая Telegram.
- [ ] **D7 (SHOULD)** Хранит и **демонстрационные** версии (упрощённые превью, картинки для слайдов) отдельно
      от «рабочих», чтобы презентация собиралась из них.
- [ ] **D8 (MAY)** Складывает всё в один скачиваемый пакет (ZIP) на финале.

## E. Статистика и финансовая модель

Агент **внутри корректно отрабатывает** статметоды «другого курса» — для построения финмодели и
обоснования решения. Должен делать это сам (Jupyter/Python), а не «на словах».

- [ ] **E1 (MUST)** Авто-EDA датасета: пропуски, выбросы, распределения, описательные статистики.
- [ ] **E2 (MUST)** **Корреляционный анализ** драйверов (матрица + интерпретация: что с чем связано).
- [ ] **E3 (MUST)** Юнит-экономика: ARPU, Churn, CAC, **LTV**, **LTV/CAC**, **Payback period**.
- [ ] **E4 (MUST)** Финмодель: трафик → конверсия → MRR/выручка → расходы → **EBITDA**, горизонт ≥ 12–24 мес.
- [ ] **E5 (MUST)** **Точка безубыточности** (breakeven) по накопленному денежному потоку.
- [ ] **E6 (MUST)** **NPV** при заданной ставке дисконтирования (+ окупаемость проекта).
- [ ] **E7 (MUST)** **Сценарный анализ**: base / optimistic / pessimistic.
- [ ] **E8 (MUST)** **Анализ чувствительности → Tornado-chart** (±X% по ключевым драйверам, ранжирование).
- [ ] **E9 (MUST)** **Монте-Карло** (≥ 4000–5000 симуляций): распределение результата, **P5/P50/P95**
      (или P10/P90), **вероятность прибыли / P(NPV>0)**.
- [ ] **E10 (MUST)** Каждый расчёт сопровождает **выводом в 1 предложение** (для слайда) и графиком (картинкой).
- [ ] **E11 (SHOULD)** Проверяет корректность входных допущений и численную устойчивость (диапазоны, клиппинг,
      сид для воспроизводимости).
- [ ] **E12 (MAY)** Проверка гипотез / A-B-сравнение вариантов, если того требует кейс слушателя.

## F. Презентация в шаблоне результатов

Финал — защита, собранная из накопленных артефактов в шаблоне заказчика.

- [ ] **F1 (MUST)** Собирает презентацию-защиту **из финальных и демо-артефактов** (D2/D7), а не заново.
- [ ] **F2 (MUST)** Верстает в **шаблоне результатов заказчика** (MARP): фирменный стиль, секционные слайды,
      крупная типографика. Шаблон (брендовый, напр. Сбер) **заранее загружается в GitLab** и подаётся агенту —
      презентация собирается внутри, без внешних инструментов (Gamma и т.п.). MARP агент ставит сам (быстро).
- [ ] **F3 (MUST)** Структура защиты: бизнес-вопрос → рынок → дифференциаторы → JTBD-инсайт → финмодель (NPV)
      → риск (Монте-Карло) → вывод и шаги. ≤ 3 буллета и ≥ 1 цифра/график на слайд.
- [ ] **F4 (MUST)** Вставляет ключевые **графики-картинки** из `assets/img/` (Tornado, NPV, Монте-Карло).
- [ ] **F5 (MUST)** Экспортирует в `.pdf` (и исходный MARP `.md`).
- [ ] **F6 (SHOULD)** Генерирует структуру 5-минутного видео-защиты (тайминг по слайдам).
- [ ] **F7 (SHOULD)** Делает финальную паузу на ревью перед сдачей на ИИ-грейдинг.

## G. Качество и самопроверка (gates)

- [ ] **G1 (MUST)** Перед фиксацией артефакта прогоняет его по критериям шага (мини-рубрика) и сообщает,
      что соответствует, а что нет.
- [ ] **G2 (SHOULD)** Не закрывает шаг, если обязательный под-артефакт отсутствует (напр. нет картинки графика).
- [ ] **G3 (SHOULD)** Держит единый сквозной контекст продукта: метрики из дерева метрик → гипотезы → драйверы
      финмодели не противоречат друг другу.
- [ ] **G4 (MAY)** Финальный «критик»: проверяет, что в защите закрыты все шаги и нет неподтверждённых утверждений.

## H. Среда исполнения и инструменты («упряжка» агента)

> Это требования к **окружению вокруг модели** (sandbox/harness), а не к промптам. Конкретная платформа
> ещё не выбрана: скорее всего это будет **собственный sandbox на базе существующего рантайма**. Поэтому
> здесь фиксируем **что среда должна уметь** — под эти требования и подбираем harness. Привязка к разделам:
> H1→B, H2–H4→E, H5–H7→D/F.

- [ ] **H1 (MUST)** Доступ в интернет и глубокий поиск (питает раздел **B**). Два продукта Perplexity:
  - **Perplexity Search API** — `POST https://api.perplexity.ai/search` → структурированный `results[]`
    (`title`, `url`, `snippet`, `date`) для собственного RAG-пайплайна.
  - **Perplexity Sonar** (chat completions) — `POST https://api.perplexity.ai/chat/completions`, ответ с цитатами:
    `sonar` · `sonar-pro` · `sonar-reasoning-pro` · **`sonar-deep-research`** (глубокое исследование сотен
    источников — под тяжёлые B-кейсы; дороже и ниже rate-limit).
  - Резерв: Tavily / Brave / SerpAPI.
- [ ] **H2 (MUST)** Песочница **Python** с состоянием между шагами (Jupyter-ядро): выполнение кода, сохранение
      вывода, **перехват графиков как PNG**, таймаут и лимит памяти. Графики агент строит **прямо в Python** —
      отдельная Jupyter-тетрадь как самостоятельный артефакт **не обязательна** (лишняя сущность для слушателя);
      ядро ставим для исполнения и состояния, а не как формат сдачи.
  > **Установка библиотек.** Полный `pip install` на лету разрешён (карт-бланш). Но базовый научный стек
  > (H3–H4, OCR/документы) **переустанавливаем при старте контейнера**, сразу разрешая конфликты версий:
  > пусть машина стартует дольше, зато у агента сразу всё развёрнуто и он не вязнет в починке версий
  > посреди занятия. Учесть: иногда библиотека не ставится из нашего сегмента сети (PyPI) — проверить заранее.
- [ ] **H3 (MUST)** Установлен научный стек, закрывающий раздел **E**: `numpy`, `pandas`, `scipy`, **`statsmodels`**,
      `scikit-learn`, `pingouin` (стат-тесты), `prophet`/`pmdarima` (временные ряды). Монте-Карло, NPV, Tornado,
      сценарии — считаются кодом, а не «на словах».
- [ ] **H4 (MUST)** Визуализация и экспорт картинок (для **E10/F4**): `matplotlib`, `seaborn`, `plotly` + `kaleido`
      (экспорт plotly в PNG).
- [ ] **H5 (MUST)** Работа с документами для раздела **D/F**: создание/редактирование/чтение
      `.docx` (`python-docx`), `.xlsx` (`openpyxl`/`xlsxwriter`), `.pptx` (`python-pptx`),
      `.pdf` (чтение `pdfplumber`/`pypdf`, генерация `reportlab`/`weasyprint`), `.md`/`.csv`.
- [ ] **H6 (MUST)** **MARP** для сборки презентации-защиты (под **F2/F5**), экспорт в PDF.
- [ ] **H7 (MUST)** Файловое хранилище артефактов + выдача **публичных ссылок** в чат (под **D6**); упаковка в ZIP (**D8**).
  > Объём форматов на старте **ограничен** (`.md`/`.doc`/`.pdf`/`.png`/`.csv` + MARP-PDF). Интерактивные
  > отчёты/презентации (HTML) пока **не делаем** — у практики меньше дня, цель прогнать весь цикл в базовых
  > форматах. Публичные ссылки на файлы — да; интерактив наружу — позже.
- [ ] **H8 (SHOULD)** RAG/поиск по документам (под **B1**, анализ транскрипций и поиск по материалам курса):
      эмбеддинги (`sentence-transformers` или embeddings-API) + векторная БД (`faiss`/`chromadb`), `tiktoken`.
- [ ] **H9 (SHOULD)** Парсинг веба и OCR: `requests`/`httpx`, `beautifulsoup4`/`lxml`/`trafilatura`,
      `Pillow`/`pytesseract`, `tabula-py` (таблицы из PDF).
- [ ] **H10 (MUST)** Секреты в переменных окружения: `PERPLEXITY_API_KEY` (один на Search API и Sonar),
      ключ основного LLM-провайдера; (опц.) `TAVILY_API_KEY`/`BRAVE_API_KEY`, `SPEECH_API_KEY` (ASR, см. **H12**).
  > **Провайдер.** MVP — на **OpenAI**. На случай требования «только отечественные сервисы» держим в уме
  > запасной путь через **LLM-роутер** (`agentplatform.ru`): там есть **GigaChat 2 Max** и **YandexGPT 5**.
  > Оговорка: у отечественных моделей сложнее с веб-поиском/Deep Research (раздел **B**) — для тяжёлых
  > исследований остаёмся на провайдере с поиском. Переключение провайдера — через конфиг, без переписывания логики.
- [ ] **H13 (SHOULD)** **Проверка доступности российских сайтов.** Песочница должна открывать ресурсы под
      сертификатом **Минцифры** (Росстат, ЦБ РФ, Госуслуги, банки) — иначе агент «не видит» отечественные
      источники для разделов **B/E**. Прогнать чек-листом ключевые госсайты; если поиск/фетч идёт изнутри через
      API провайдера — проверить, что он не упирается в недоверенный сертификат. (Perplexity, по наблюдениям,
      открывает; OpenAI — под вопросом, т.к. блокирует РФ.)
- [ ] **H11 (MAY)** Доступ к API открытых данных: Росстат, World Bank, ЦБ РФ — для B/E, где нужны реальные ряды.
- [ ] **H12 (SHOULD)** **Обработка голосовых сообщений.** Участник шлёт voice в Telegram (`.ogg`/Opus) →
      агент **транскрибирует** и дальше работает с текстом как обычно. Требование — сильное распознавание
      **русской речи** с пунктуацией и нормализацией чисел. Это «вау»-интерфейс на демо (общение голосом),
      поэтому делаем. Правило поведения прописывается прямо в **AGENT.md / «конституции» агента**: «если
      пришло голосовое — сначала транскрибируй, потом действуй».
  > **Основной путь — через LLM-роутер.** На роутере **Whisper доступен** (подтверждено на встрече) —
  > подключаем транскрипцию через него (тот же роутер, что и для текстовых моделей), отдельный ASR-провайдер
  > не нужен. Если на роутере есть OpenAI-совместимый `transcribe` — используем его (`SPEECH_API_KEY`/конфиг
  > роутера, см. **H10**).
  - Резерв (если путь через роутер недоступен): **Whisper `large-v3`** / `faster-whisper` локально в песочнице
    **H2** (контейнер ~6 ГБ), **Yandex SpeechKit v3** или **SberSaluteSpeech/GigaAM** — для топового русского.

**Среда (минимум):** Python ≥ 3.11, `jupyter`/`ipykernel`/`nbclient`/`nbformat`, менеджер пакетов `pip`/`uv`.

### Минимальный сценарий приёмки (smoke-test для разработчика)

1. Слушатель в Telegram: «начинаем практику по продукту X» → агент берёт план, объявляет шаг 0 (**A1–A4**).
2. Агент строит дерево метрик, сохраняет `s0_metric_tree_v1` (**D1–D6**), показывает по ссылке, ждёт ревью (**C1**).
3. Слушатель правит → `v2`, старая версия цела (**C2, D3**), агент идёт к шагу 1 только после «ок» (**C3**).
4. На шаге исследования агент ищет в вебе (Perplexity Search/Sonar), разводит факт/гипотезу, не выдумывает цифры (**B1–B4, H1**).
5. На шаге финмодели агент сам считает NPV, Tornado и Монте-Карло в Python-песочнице, отдаёт графики-картинки и выводы (**E, H2–H4**).
6. На финале собирает MARP-защиту из накопленных артефактов в шаблоне заказчика, экспортирует PDF (**F, H5–H6**).
7. По запросу «собери всё» отдаёт пакет со всеми артефактами и `manifest.json` (**D4, D8, H7**).

Если все 7 пунктов проходят — агент готов к пилоту на одном курсе.
