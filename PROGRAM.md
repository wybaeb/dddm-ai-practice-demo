# Программа сквозной практики (план занятия для агента)

7 этапов: от бизнес-вопроса до защиты. Логика описания — **этап → результат (артефакт) → что делает
агент → детальный промпт**. Промпты — рабочие, взяты из реального курса DDDM (модули предматериалов,
М1, М2) и адаптированы под демо-продукт «Пульс» (умные часы с проактивным ИИ-агентом). Слушатель может
подставить свой продукт — структура промптов сохраняется.

После **каждого** этапа — пауза на ревью: агент показывает результат и ждёт «утверждаем / правим»,
дальше идёт только после согласования. Поведенческая спецификация агента (для разработчика) — отдельно,
в [CHECKLIST.md](./CHECKLIST.md); её **не нужно** показывать слушателю как «требования к агенту» — для
слушателя это этапы с конкретным результатом.

## Этап 0 — Дерево метрик продукта · Предматериалы

**Результат:** markdown-файл с деревом метрик в фенс-блоке ```` ```mermaid ```` (главная →
расчётные → опережающие), с проставленными типами `leading / calc / lagging` и формулами расчётных.

**Зачем:** разложить главную метрику результата на управляемые драйверы. Это база для гипотез,
эксперимента и финмодели — без дерева не выбрать, на какой рычаг нажать.

**Что делает агент:**
1. Уточняет продукт, аудиторию и бизнес-цель квартала.
2. Помогает определить **одну** главную (запаздывающую) метрику и обосновать выбор.
3. Раскладывает её на 3–4 опережающих (управляемых) метрики-драйвера.
4. Добавляет расчётные (промежуточные) метрики и **формулы** связи.
5. Рендерит дерево в `mermaid graph TD` (главная сверху, опережающие — листья), сохраняет `.md` (+ `.png`).
6. Пауза: «проверь типы метрик и формулы связей — что правим?»

**Промпт (пошагово, копировать в DeepSeek / ChatGPT / Claude / GigaChat):**
```
Ты — продуктовый аналитик. Построй дерево метрик для продукта вместе со мной, по шагам.
Продукт: умные часы с проактивным ИИ-агентом «Пульс». Монетизация: продажа устройства + платная
подписка на ИИ-функции. Аудитория: те, кто следит за здоровьем и продуктивностью.
Бизнес-цель квартала: рост выручки от подписки за счёт удержания, без роста стоимости привлечения.

Шаг 1. Предложи ОДНУ главную (запаздывающую, lagging) метрику результата и обоснуй в 1–2 предложениях,
       почему именно она отражает ценность и бизнес-цель квартала.
Шаг 2. Разложи её на 3–4 опережающих (leading) метрики — то, чем можно управлять сегодня.
Шаг 3. Добавь расчётные (calc) метрики между ними и ФОРМУЛЫ связи (как из опережающих собирается
       расчётная, а из расчётных — главная).
Шаг 4. Собери дерево в виде кода mermaid `graph TD`: главная сверху, расчётные в середине,
       опережающие — листья. В подписях узлов укажи тип: например `Retention D7 (leading)`,
       `Churn (calc)`, `MRR (lagging)`.
Формат ответа: (1) главная метрика + обоснование; (2) список метрик с типами и формулами расчётных;
(3) валидный блок ```mermaid```. Не выдумывай данные — только структуру, типы и формулы.
```

## Этап 1 — Бизнес-вопрос и исследование рынка · Модуль 1.2

**Результат:** обзор рынка с разметкой «факт / интерпретация / гипотеза», лонглист «Проблема × Сегмент»,
Product Vision (шаблон Мура) и Lean Canvas (`.doc`/`.md`).

**Зачем:** сформулировать вопрос исследования и понять, где рынок и узкие сегменты с готовностью платить.

**Что делает агент:**
1. Фиксирует 2–3 ключевых вопроса исследования.
2. Запускает обзор рынка через веб-поиск / Deep Research по детальному промпту.
3. Требует ссылку+цитату на каждый факт и **самокритику** модели в конце.
4. Просит слушателя проверить 1–2 источника вручную (правило The Mom Test: факт ≠ оценка).
5. Собирает лонглист и Lean Canvas. Пауза на ревью приоритезации сегментов.

**Промпт (assistant с веб-поиском — Perplexity / ChatGPT / Claude / GigaChat / DeepSeek):**
```
Ты — продуктовый аналитик. Проведи исследование рынка для продукта:
умные часы с проактивным ИИ-агентом «Пульс»; сегмент — те, кто следит за здоровьем и
продуктивностью; гипотеза ценности — платят за проактивность (подсказки), а не за датчики.

Сделай:
1. Объём и динамика рынка (TAM/SAM/SOM, если есть данные), ключевые тренды.
2. Реализована ли уже эта идея? Кто это делает и как?
3. Перспективные узкие сегменты (ниши) с высокой готовностью платить.
4. Аргументы «за» и «против» запуска.

Требования к ответу:
- К каждому факту — ссылка на источник и точная цитата.
- Раздели факты, интерпретации и гипотезы (помечай каждый блок).
- Не выдумывай. Если данных нет — напиши «нет данных».
- В конце дай самокритику: где вывод слабый и что стоит перепроверить вручную.
```
> После ответа модели: открыть 1–2 источника на каждый ключевой вывод и проверить вручную.

## Этап 2 — Конкурентный анализ (FCM) · Модуль 1.3

**Результат:** Feature Comparison Matrix (продукт × конкуренты, с разметкой гигиена/дифференциатор и
источником на каждую ячейку) + очерченный скоуп MVP «один дифференциатор + минимум гигиены».

**Зачем:** отделить «гигиену» (есть у всех) от дифференциаторов и выбрать честную метрику бенчмаркинга.

**Что делает агент:**
1. Собирает список прямых и **косвенных** конкурентов и продуктов-заместителей (как закрывают
   потребность сейчас).
2. Строит матрицу по функциям, подтверждая каждую ячейку ссылкой и цитатой.
3. Размечает каждую функцию: **гигиена** или **дифференциатор**; помечает кандидаты в MVP.
4. Фиксирует одну ключевую метрику для бенчмарка и обоснование.
5. Пауза: «согласен с дифференциаторами и скоупом MVP? что добавить?»

**Промпт:**
```
Собери Feature Comparison Matrix для продукта «умные часы с проактивным ИИ-агентом „Пульс“».
Конкуренты: укажи 3–4 прямых, добавь косвенных и продукты-заместители (чем человек закрывает
потребность сейчас — телефон, фитнес-браслет, ручной трекинг).

Строки — функции (точность пульса, сон, проактивные голосовые подсказки, голосовой агент,
автономность, цена подписки и т.п.). Колонки — ваш продукт + конкуренты + заместитель.
Для каждой функции:
- отметь наличие у каждого игрока: да / нет / частично;
- классифицируй функцию: [ГИГИЕНА] (есть у всех, ожидаемо) или [ДИФФЕРЕНЦИАТОР] (выделяет нас);
- пометь, входит ли в MVP;
- подтверди каждую ячейку ссылкой на источник и точной цитатой.
Раздели факты и предположения. Не выдумывай — нет данных, пиши «нет данных».
В конце: одна ключевая метрика для честного бенчмарка и почему; и скоуп MVP
(один уверенный дифференциатор + минимум гигиены, остальное — в бэклог).
```

## Этап 3 — Интервью и JTBD · Модуль 1.4

**Результат:** опросник Switch-interview/JTBD (45–60 мин), транскрипция (живая или синтетическая),
разбор: топ-5 Jobs to be Done, 4 силы переключения (цитатами), As-is CJM, 3–5 проверяемых гипотез.

**Зачем:** вытащить «работы», боли и силы переключения — на чём держится спрос и удержание.

**Что делает агент:**
1. **Готовит опросник** по JTBD (промпт 3a) — реконструирует цепочку переключения, про наш продукт
   спрашивает только в конце.
2. Если нет живых респондентов — **генерирует синтетическую транскрипцию** (промпт 3b, с дисклеймером
   «синтетика ≠ живое интервью»). Кто может — проводит 2–3 реальных интервью.
3. **Разбирает транскрипцию** (промпт 3c): Jobs, 4 силы цитатами, CJM, гипотезы.
4. Пауза на проверку инсайтов против реальных слов респондента.

**Промпт 3a — генерация опросника JTBD:**
```
Ты — ведущий UX-исследователь, специалист по Jobs-to-be-Done (Switch interview, школа Боба Мёсты).
Сгенерируй опросник для глубинного интервью, который раскроет, на какую «работу» пользователь нанимает
продукты в этой категории и какие силы двигают/удерживают его при переключении.

КЛЮЧЕВЫЕ ПРИНЦИПЫ JTBD (соблюдать строго):
1. Job — устойчивая потребность, не функция: «глагол + объект + контекст» (напр. «контролировать пульс
   во время утренней пробежки»).
2. Продукт нанимается на работу, а не покупается; один пользователь нанимает РАЗНЫЕ продукты на РАЗНЫЕ работы.
3. Реконструируй всю цепочку переключения в хронологии: Старое решение → Триггер(Push) → Поиск →
   Новое решение → Что дальше. Не спрашивай «что нравится в нашем продукте» в лоб.
4. Четыре силы (Forces of Progress): PUSH (что выталкивает из старого), PULL (что притягивает к новому),
   HABIT (инерция, привычка), ANXIETY (страхи перед новым). Цель — 3–5 цитат на каждую силу.
5. Собирай все три типа работ: функциональные, эмоциональные, социальные.
6. Спрашивай про КОНКРЕТНЫЕ ЭПИЗОДЫ («расскажи последний раз, когда…»), а не про мнения.
7. Вопросы про НАШ продукт — только в самом конце.

Вход: продукт — «Пульс» (умные часы с проактивным ИИ-агентом); сегмент — [укажи сегмент].

Алгоритм:
ШАГ 1. Сформулируй топ-5 работ (≥3 функциональные + 1 эмоциональная + 1 социальная), объясни различия.
ШАГ 2. По каждой работе — блок: (2.1) установка контекста; (2.2) текущее решение (с датировкой «как
       давно/как часто»); (2.3) прошлое решение и переход — ядро Switch-interview: триггер, Push, Pull,
       Habit, Anxiety, кто повлиял, длина воронки переключения; (2.4) эмоции и социальный контекст;
       (2.5) будущее идеальное решение и условия перехода; (2.6) anti-job (чего НЕ хочет).
ШАГ 3. ТОЛЬКО после всех 5 работ — секция про наш продукт: знакомство; разбор по 4 силам; реконструкция
       гипотетической первой покупки (первые 3 шага и точки потери).
ШАГ 4. Завершение: «30 секунд про идеальное решение коллеге», «о чём я не спросил», запрос на повторный контакт.
ШАГ 5. Техобвязка: длительность 45–60 мин; инструкция интервьюеру (не задавать наводящих, не предлагать
       решений, углублять оценки до эпизодов, продукт — только в Шаге 3, 2–3 истории на каждую силу).
Формат вывода: # Опросник JTBD для «Пульс» → Топ-5 работ → блоки по Job 1..5 → блок про наш продукт →
завершение → инструкция интервьюеру.
```

**Промпт 3b — синтетическая транскрипция (если нет живых респондентов):**
```
Ты — представитель целевой аудитории «Пульс», тебя пригласили на глубинное интервью. Ответь максимально
достоверно на ВСЕ вопросы опросника так, как ответил бы реальный человек из сегмента [сегмент].
Сначала придумай конкретную непротиворечивую персону (имя, возраст, профессия, образ жизни, какие
устройства уже использует) — 3–4 строки — и дальше отвечай строго от её лица.

Правила достоверности:
1. Отвечай по КОНКРЕТНЫМ ЭПИЗОДАМ («в прошлый вторник на пробежке…»), а не оценками. Рассказывай истории.
2. Будь реалистичен и противоречив, как живой человек (где-то путаешься, не помнишь точную дату).
3. Опирайся на типичное поведение сегмента и открытую статистику.
4. Показывай эмоции и социальный контекст, где уместно.
5. НЕ подыгрывай продукту: если функция не нужна — так и скажи. Барьеры и тревоги важны не меньше восторгов.
6. На каждый вопрос — отдельный развёрнутый ответ (2–6 предложений), без воды.
Опросник:
<questionnaire>
[ВСТАВИТЬ ОПРОСНИК ИЗ ПРОМПТА 3a]
</questionnaire>
Формат: диалог «Интервьюер: … / Респондент (<имя>): …» по всем вопросам по порядку. В конце — «— конец интервью —».
```
> ⚠️ Синтетика — тренажёр цикла (опросник → транскрипт → анализ) и образец формата, **не замена** живым
> интервью: модель не вскрывает по-настоящему новых инсайтов и может галлюцинировать детали. Для решений
> о продукте — 2–3 живых интервью.

**Промпт 3c — разбор транскрипции:**
```
Проанализируй транскрипцию глубинного интервью (вставлю ниже).
Выдели:
1) Топ-5 Jobs to be Done (формулировка «когда…, я хочу…, чтобы…»).
2) 4 силы переключения: Push, Pull, Habit/Inertia, Anxiety — ДОСЛОВНЫМИ цитатами из текста (2–3 на силу).
3) As-is CJM: этапы текущего решения боли.
4) 3–5 ПРОВЕРЯЕМЫХ гипотез в виде «если [действие], то [метрика] изменится на [порог], потому что …».
Опирайся ТОЛЬКО на сказанное; вывод без опоры на цитату — удали (это домысел). Где домысливаешь — помечай
[ИНТЕРПРЕТАЦИЯ]. Если данные синтетические — отметь это (влияет на доверие к выводам).
ТРАНСКРИПЦИЯ:
"""<сюда агент подставляет транскрипцию>"""
```

## Этап 4 — Внешние факторы + аналитическая записка · Модуль 1.5–1.6

**Результат:** аналитическая записка из 6 разделов (`.pdf`/`.md`) с явной разметкой
факт/интерпретация/гипотеза + **таблица валидации выводов** в приложении.

**Зачем:** свести рынок, конкурентов, JTBD и внешние факторы в одно решение и отфильтровать
непроверенные утверждения перед финмоделью.

**Что делает агент:**
1. Строит матрицу внешних факторов: сезонность, конкуренты, макро, алгоритмы платформ, регуляторика.
2. Подтягивает артефакты этапов 1–3 из памяти, собирает записку из 6 разделов.
3. **Прогоняет выводы через валидацию** (промпт 4b): корреляция ≠ причинность, конфаундеры, как проверить.
4. Прикладывает таблицу валидации (вывод / тип / способ проверки / статус: подтверждён / отклонён / под вопросом).
5. Экспорт `.pdf`, пауза на финальное ревью.

**Промпт 4a — сборка записки:**
```
Собери аналитическую записку (1–2 стр.) из наших артефактов. Структура — ровно 6 разделов:
1. Бизнес-вопрос и рынок
2. Конкуренты и дифференциаторы (из FCM)
3. JTBD и силы переключения (из интервью)
4. Матрица внешних факторов (сезонность / конкуренты / макро / алгоритмы платформ / регуляторика)
5. Проверяемые гипотезы (каждая — с метрикой проверки)
6. Вывод и рекомендация к финмодели
В каждом пункте маркируй [ФАКТ+источник] / [ИНТЕРПРЕТАЦИЯ] / [ГИПОТЕЗА]. Тон — сухой, для принятия решения.
```

**Промпт 4b — критическая проверка выводов (валидация):**
```
Вот мои выводы по исследованию: [вставить выводы].
Проверь их строго:
1. Раздели на факты, интерпретации и гипотезы. Для каждого факта потребуй источник.
2. Где я мог перепутать корреляцию с причинностью? Назови возможные: скрытые общие причины,
   обратную причинность, сезонность.
3. Какие внешние факторы могли повлиять (конкуренты, алгоритмы платформ, макроэкономика, регуляторика)?
4. Для ключевой опережающей метрики: какой нужен размер выборки и логика эксперимента (контрольная
   группа, A/B), чтобы вывод был валиден?
Будь критичен. Не подтверждай мои выводы из вежливости — ищи слабые места.
```

## Этап 5 — Данные, EDA и финмодель · Модуль 2

**Результат:** трассируемый EDA датасета (распределения, выбросы по IQR, корреляции, сезонность,
воронка) + 3-летняя финмодель с NPV, Tornado-чувствительностью и Монте-Карло; графики — картинками (PNG),
каждый с выводом в 1 предложение для слайда.

**Зачем:** ответить на бизнес-вопрос цифрами — стоит ли запускать и какой драйвер решает.

**Что делает агент:**
1. Считает **сам** в Python (рабочее пространство с исполнением кода), а не «на словах». Графики строит
   прямо в Python и сохраняет картинками (отдельная Jupyter-тетрадь как сущность не обязательна).
2. Авто-EDA по детальному промпту, затем unit-экономика и 3-летняя модель доходов/затрат.
3. NPV при ставке дисконтирования, Tornado по драйверам, Монте-Карло (≥4000 прогонов) → P10/P50/P90, P(NPV>0).
4. Пауза: «какие допущения меняем перед защитой?» (см. интерактивный блок на странице).
