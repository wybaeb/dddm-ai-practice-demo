# Инструкция для коллег

Демо-набор «Сквозная практика с ИИ-ассистентом» (DDDM). Две роли — выберите свою.

- **Ссылка (GitHub Pages):** https://wybaeb.github.io/dddm-ai-practice-demo/
- **Пароль:** `dddm2026` (вводится один раз, дальше браузер запоминает)
- Можно и без интернета: открыть присланный файл `dddm-demo.html` двойным кликом в браузере.
- 🔓 **Ссылку для ИИ-агента** (`AGENT.md`) можно давать **без пароля** — она лежит в открытой части
  парольной страницы; агент читает программу из публичного репозитория, расшифровка ему не нужна.

> Аудитория практики — **слушатели-руководители** (топ-менеджеры организаций).

---

## ⚙️ Шаг 0 (один раз, для любого, у кого есть доступ к репозиторию)

Чтобы ссылка Pages заработала, надо один раз включить публикацию:

1. Открыть репозиторий **github.com/wybaeb/dddm-ai-practice-demo** → **Settings** → **Pages**.
2. В «Build and deployment» → Source: **Deploy from a branch**.
3. Branch: **main**, папка **/(root)** → **Save**.
4. Через 1–2 минуты страница откроется по адресу выше.

Пока тумблер не включён — пользуйтесь присланным файлом `dddm-demo.html` (он самодостаточный).

## 👀 Роль 1 — Ознакомиться (заказчик / методист / руководитель)

**Цель:** посмотреть формат и оценить, то ли мы делаем. Ничего настраивать не нужно.

1. Откройте ссылку (или файл), введите пароль `dddm2026`.
2. Пройдите разделы сверху вниз:
   - **Как работает** — принцип «агент = наставник»: промпт → выполнение → результат → пауза на ревью.
   - **Этапы** — раскройте любой из 7 этапов: вверху каждого — **результат (артефакт)**, ниже — что делает
     агент и ключевой промпт. Это ядро страницы: смотрим, тот ли результат даёт каждый этап.
   - **Финмодель** — подвигайте слайдеры (конверсия, ARPU): NPV, Tornado и Монте-Карло пересчитываются вживую.
   - **Защита** — пролистайте финальную презентацию в нашем шаблоне (стрелками).
3. **Что прислать в ответ:** где промахнулись по составу этапов и результатов, чего не хватает, что лишнее.

> Полная инженерная спецификация агента (что он обязан уметь) — в `CHECKLIST.md` в репозитории. На странице
> для слушателя это намеренно **не** показывается как «требования к агенту» — там этапы с результатами.

⏱ Занимает ~10–15 минут. Технических знаний не требуется.

## 🧪 Роль 2 — Протестировать работоспособность агента (разработчик / приёмка)

**Цель:** проверить, что реальный агент-исполнитель соответствует чек-листу.

1. Откройте **`CHECKLIST.md`** в репозитории — это спецификация проверяемых способностей
   (MUST / SHOULD / MAY), сгруппированных A–H (включая раздел **H** — среда исполнения и «упряжка» агента).
2. Возьмите тестовый продукт: либо демо-кейс «Пульс» (умные часы с ИИ), либо свой.
3. Прогоните **smoke-test из 7 шагов** (в конце `CHECKLIST.md`): дайте агенту команду
   «начинаем практику по продукту …» и идите по шагам.
4. На каждом шаге сверяйтесь с **MUST**-пунктами:
   | Группа | Что проверяем у агента |
   |---|---|
   | **A** | держит план в памяти, знает текущий шаг, не перескакивает |
   | **B** | проводит исследование, разводит факт/гипотезу, **не выдумывает данные** |
   | **C** | делает паузу на ревью, переделывает по правкам, дальше — только после «ок» |
   | **D** | сохраняет **все** артефакты (исходные/промежуточные/финальные/демо/картинки) с версиями |
   | **E** | сам считает EDA, корреляции, NPV, Tornado, Монте-Карло (а не «на словах») |
   | **F** | собирает защиту из артефактов в шаблоне результатов, экспорт PDF |
   | **G** | самопроверка по рубрикам, сквозная непротиворечивость |
   | **H** | среда исполнения: pip-карт-бланш + переустановка библиотек на старте, MARP, голосовые через LLM-роутер, доступность сайтов РФ (Минцифры), публичные ссылки на артефакты |
5. Отмечайте выполнение прямо в **`CHECKLIST.md`** (галочки `- [ ]` / `- [x]`) — по группам A–H и
   уровням MUST/SHOULD/MAY.
6. **Что прислать в ответ:** список дефектов — какой пункт (например `E9`) не выполнен и как воспроизвести.

**Критерий приёмки:** все 7 пунктов smoke-test проходят и MUST = 100% → агент готов к пилоту на одном курсе.
